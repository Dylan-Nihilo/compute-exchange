# 05 · 功能需求 · AI Token 工厂（独立 newapi）

> 需求编号：`REQ-C-xxx`。
> 决策 D3：**新部署一套独立 newapi 实例**（与现有 kaizo.top 物理/逻辑隔离），主平台负责账户、下单、支付、导流与额度同步；**计量计费在 newapi 内完成**，不在主平台重写。

## 5.1 定位与边界

| 项 | 说明 |
|----|------|
| Token 工厂本体 | 独立部署的 newapi 实例（API 网关 + 计量 + 令牌管理 + 渠道管理） |
| 主平台职责 | 商品/套餐展示、下单、易宝支付、用户账户与 newapi 账户映射、服务额度发放指令、用量账单聚合展示、API Key 下发 |
| newapi 职责 | OpenAI 标准 API 封装、Token 精准计量、令牌(API Key)管理、上游模型渠道与倍率、用量统计 |
| 业务模式 | ① 代销：采购大厂合规模型 Token 池零售；② 包销：锁定某机房推理算力自部署开源模型独家售卖；③ 部署技术服务：一次性部署费 + 运维年费 |

> 关键：v1 通过 **newapi 既有的 Admin/管理 API** 完成"建用户/发放 quota/建令牌/查用量"，避免二次开发计量系统。具体接口能力以新部署 newapi 版本为准（移交架构阶段核对）。

## 5.2 模型与套餐管理

- **REQ-C-001 (P0)** 模型上架：运营在主平台配置可售模型（名称、厂商、上下文长度、单价倍率、适用场景、是否开源自部署/代销），与 newapi 渠道对应。
- **REQ-C-002 (P0)** Token 套餐：定义套餐（如 ¥X = N Token / 包月不限量某模型 / 阶梯价），关联模型与额度。
- **REQ-C-003 (P0)** 合规标识：自部署开源模型须标注算法备案号/状态（见第11章），未备案模型不得对公众开放。
- **REQ-C-004 (P1)** 阶梯/折扣定价、企业询价。
- **REQ-C-005 (P1)** 包销算力来源标注：包销模式下标注承载机房（内部可见），用于成本核算。

## 5.3 账户与额度打通

- **REQ-C-010 (P0)** 账户映射：主平台用户首次购买 Token 时，在 newapi 创建/绑定对应用户，建立映射关系（主平台 userId ↔ newapi userId）。
- **REQ-C-011 (P0)** 发放服务额度：支付成功回调后，调用 newapi 管理接口为该用户发放 quota，失败重试 + 告警 + 对账。
- **REQ-C-012 (P0)** API Key 下发：在 newapi 为用户创建/获取令牌(API Key)，连同 base_url、调用示例在主平台展示。
- **REQ-C-013 (P0)** 额度与用量回显：主平台聚合展示用户当前额度、已用量、调用明细（从 newapi 拉取或同步）。
- **REQ-C-014 (P1)** 令牌管理：用户可在主平台重置/禁用/新建 API Key（代理调用 newapi）。
- **REQ-C-015 (P1)** 用量预警：额度低于阈值提醒购买套餐；可设自动续费（P2）。

## 5.4 计量与账单

- **REQ-C-020 (P0)** 用量账单：实时/准实时展示 Token 消耗（按模型、按时间），明细可导出。统一分词/计量口径以 newapi 为准，避免计量纠纷。
- **REQ-C-021 (P0)** **计量独立链路**：Token 业务的财务、台账与算力租赁业务**完全分开**（业务输入硬要求），账单与对账独立。
- **REQ-C-022 (P1)** 发票：Token 消费可单独申请发票（与算力订单发票体系统一，但科目区分）。

## 5.5 部署技术服务（一次性 + 运维）

- **REQ-C-030 (P1)** 部署服务工单：客户提交"开源大模型一键部署"需求（模型、规格、推理框架 vLLM/TensorRT-LLM、目标算力），生成服务工单。
- **REQ-C-031 (P1)** 报价与合同：一次性部署费 + 运维年费报价，转订单/合同（走易宝支付 + 电子合同）。
- **REQ-C-032 (P2)** 标准化部署包：沉淀标准模型部署模板（阶段3的"自有 Token 品牌包销产品线"雏形）。

## 5.6 与主平台集成方式

- **REQ-C-040 (P0)** 单点登录/账户互信：用户在主平台无感使用 Token 工厂（SSO 或 token 互信），不重复注册。见已决策问题 Q-07（主平台封装关键操作）。
- **REQ-C-041 (P0)** newapi 实例隔离：独立域名/独立数据库/独立渠道密钥，与 kaizo.top 不共享数据。
- **REQ-C-042 (P1)** 导流：门户 Token 板块 → 套餐页 → 下单 → 控制台（newapi 或主平台封装的控制台）。

## 5.7 风控与合规
- **REQ-C-050 (P0)** 内容合规：自部署模型须有内容安全机制（关键词/拦截），满足生成式 AI 安全要求（第11章）。
- **REQ-C-051 (P0)** 滥用防护：异常高频/异常用量限流、封禁（newapi 能力 + 主平台策略）。
- **REQ-C-052 (P1)** 备案展示：对外页面在显著位置标注大模型备案号并链接官方公示（合规义务）。

## 5.8 待确认（移交架构阶段）
- 新 newapi 版本与可用管理 API 清单（发放 quota/建令牌/查用量是否齐全）→ 架构阶段核对。
- 主平台与 newapi 的账户映射和接口边界（Q-07）。
- 代销上游（大厂模型 Token 池）采购渠道与合规授权（第12章）。

## 5.9 验收要点
1. 用户在主平台购买 Token 套餐 → 支付成功 → newapi 自动发放服务额度 → 拿到 API Key → 成功调用模型 → 主平台看到用量账单。
2. Token 财务台账与算力业务台账物理分离。
3. 独立 newapi 与现有 kaizo.top 无数据耦合。
