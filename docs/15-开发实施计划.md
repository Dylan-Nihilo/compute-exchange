# 15 · 开发实施计划

> 编制日期：2026-07-13  
> 依据：`需求方案v1.1（区块链+算力）.pdf`、`项目功能清单.zip` / `docs/01-14`、`wireframe.html`、已批准的前端原型设计文档与当前代码。  
> 当前目标：先交付可验收的前端高保真原型，再沿同一领域模型接入真实后端和第三方服务。

## 15.1 材料关系与执行优先级

1. `docs/superpowers/specs/2026-07-11-frontend-prototype-design.md` 中已拍板的决策优先，包括纯订单直付、延迟分账、三段式合规开关和当前阶段只做前端原型。
2. `docs/01-14` 是功能、状态机、合规和非功能需求的详细基线。
3. PDF 是业务汇报版，适合确认目标和边界，不替代详细验收项。
4. `wireframe.html` 是信息层级和关键交互参考，不是完整页面清单，也不是可直接移植的实现。

开发前必须先修正文档中的旧编号和余额体系冲突：平台不提供现金余额充值；Token 额度仍保留，二者不可混用。

## 15.2 当前代码基线

仓库已具备 Next.js 15、React 19、TypeScript strict、Tailwind CSS v4、HeroUI v3、TanStack Query、Zustand 和 Zod。`pnpm check` 已统一类型检查、Lint 和生产构建。

当前业务实现只有启动页及粒子动画。尚未建立业务路由、领域类型、Mock service、权限守卫、表单、工作台、交易流程和端到端测试。因此应从共享契约和可运行的业务闭环开始，不能按页面逐张孤立堆砌。

## 15.3 线框稿覆盖与缺口

线框稿提供了 13 个代表页面：门户、算力市场、商品详情、确认订单、买家首页/订单详情、供给方首页/发布商品、Token 工厂、融资留资、运营审核、风控工作台。

正式开发还需补齐：

- 登录、注册、个人实名、企业认证、身份申请与切换；
- 买家订单列表、Token 管理、账单、合同、发票、消息与工单；
- 供给方资质、商品列表、履约回填、订单、结算与信用档案；
- 设备厂商、资方轻量工作台；
- 设备/施工线索、CRM、佣金台账；
- 运营端订单、资金、Token、CMS、用户/RBAC、系统设置与合规开关；
- loading、empty、error、权限拒绝和高风险操作确认状态。

线框稿中的 `login`、`register`、`equipment`、`construction` 没有对应页面，`token` / `finance` 与实际页面 ID 也不一致。正式路由按本文规划建立，不复用这些临时跳转名。

## 15.4 交付策略

### 阶段 A：高保真前端原型

目标是用 Mock 数据跑通角色、审核、上架、交易、履约、分账展示、Token 和风控闭环。页面通过 hooks 访问 service 接口，后续接 API 时只替换 service 实现。

推荐顺序：运营后台 → 多角色工作台 → 交易所 → Token/居间 → 门户。后台先行可以先建立审核、商品和订单状态机，为其余页面提供可操作的数据源。

### 阶段 B：生产系统接入

原型验收后再建设真实 API、数据库和第三方集成。易宝、KYC、电子签、newapi、BSN-DDC 都放在适配器边界后，并先接沙箱。任何第三方故障不得阻塞订单主状态保存。

## 15.5 阶段 A 任务拆分

### A0 · 需求冻结与前端契约

交付：

- 修正文档旧编号、删除现金余额充值需求，冻结 Must / Should；
- 建立 Account、Qualification、ComputeProduct、Order、SplitRecord、TokenAccount、Lead、RiskAlert、Attestation、ComplianceGate 类型；
- 定义订单、审核、商品、分账、线索状态机及合法转换；
- 建立路由清单、权限矩阵和统一状态文案；
- 落地已确认的中性主题 token：按 background / surface / text / border / action / status 定义语义颜色，业务组件不写死品牌色。

完成门槛：每个 Must 需求都能映射到页面、动作、状态机和验收用例；无 `TBD` 阻塞项。

### A1 · 应用骨架、身份与 Mock 引擎

交付：

- `(portal)`、`market`、`auth`、`console`、`admin` 路由组与共享外壳；
- 登录/注册/KYC/企业认证流程和多身份切换；
- 权限守卫、未授权页、导航与面包屑；
- 可重置、可持久化、可注入延迟/错误的 Mock service；
- 统一 loading、empty、error、toast、确认弹窗和表单校验。

完成门槛：四类用户和管理员能进入正确工作区；越权路由不可访问；刷新后 Mock 状态可恢复。

### A2 · 运营后台

交付：

- 机房/商品/厂商/内容审核队列，含补件、驳回原因和审核历史；
- 商品、订单、资金与分账规则管理；
- CRM、Token 套餐、CMS、用户/RBAC；
- 风控告警、冻结/解冻、误报和证据时间线；
- 三段式合规开关与全量审计日志。

完成门槛：审核结果能实时影响供给方和前台；高风险动作二次确认并留痕；合规开关能控制下单入口与限额。

### A3 · 多角色工作台

交付：

- 买家：订单、交付确认、退款/工单、Token、账单、合同与发票；
- 供给方：资质、商品、订单履约、凭证回填、结算和信用档案；
- 厂商：设备发布、线索、报价状态和佣金台账；
- 资方：融资线索查看、认领和跟进状态；
- 通用消息、待办和身份切换。

完成门槛：管理员审核 → 供给方上架 → 买家可见的跨角色数据联动完整；各角色只看到授权数据。

### A4 · 算力交易闭环

交付：

- 商品检索、筛选、排序、比价和信用存证入口；
- 商品详情、可售时段、余量、阶梯价和费用试算；
- 合规承诺、下单、余量锁定、模拟支付；
- 履约凭证、买家签收、超时确认、退款/争议状态；
- 延迟分账、对账、合同和发票展示。

完成门槛：至少一条订单从上架到完成全程可演示；重复提交不会生成重复订单；余量、金额和状态在各端一致。

### A5 · Token、居间、融资与门户

交付：

- Token 模型/套餐、购买、额度、API Key、用量和账单；
- 设备/施工需求登记、CRM 流转和佣金台账；
- 融资租赁展示、合规声明、留资和资方认领；
- 门户五大板块、内容页、留资入口、备案信息和存证验证入口；
- SEO 元数据、响应式布局和基础可访问性。

完成门槛：Token 购买到用量账单闭环可演示；所有门户线索进入 CRM；移动端无关键流程阻断。

### A6 · 原型验收

必须通过：

1. 审核 → 上架 → 下单 → 支付 → 履约 → 签收 → 延迟分账展示；
2. 多身份切换与路由权限；
3. 合规开关“仅展示 / 限额撮合 / 全量”三态；
4. Token 套餐购买 → 额度 → Key → 用量账单；
5. 每个核心列表/详情的 loading、empty、error；
6. 320px、768px、桌面宽度及键盘操作检查；
7. `pnpm check` 和 Playwright 核心路径全部通过。

## 15.6 阶段 B 生产化任务

### B0 · 后端基础

- 数据模型、迁移、鉴权、RBAC、对象存储、审计日志；
- OpenAPI 契约与前端 service HTTP 实现；
- 幂等键、事务边界、队列、定时任务和通知；
- 开发/测试/预发布/生产环境及密钥管理。

### B1 · 账户、资质与交易核心

- 真实注册登录、KYC/企业认证和机房资质台账；
- 商品、库存时段、余量锁定、订单与履约；
- 合同、发票、工单和争议仲裁；
- CRM、佣金和多角色数据隔离。

### B2 · 支付与 Token 集成

- 易宝二级商户进件、合单支付、延迟分账、退款、回调和日对账；
- newapi 账户映射、额度充值、Key、用量和账单；
- 所有回调验签、幂等、重放保护和人工补偿工具。

真实资金只走易宝持牌账户。平台不设现金余额，不接触或沉淀用户资金。

### B3 · 区块链与风控

- 订单创建、交付、完成、违规和仲裁结论的异步 hash 存证；
- 签名、TX ID 回写、重试、死信和公开验证；
- 挖矿排除、异常订单、刷单关联、冻结和申诉；
- 原始隐私与商业数据不上链，只保存可验证指纹。

### B4 · 合规联调与上线

- 等保三级整改、数据分级、备份恢复、漏洞与压测；
- EDI/ICP、算法与区块链备案状态接入合规开关；
- 易宝/newapi/存证服务沙箱及生产联调；
- 仅展示 → 小额限量 → 全量的灰度发布和回滚演练。

完成门槛以 `docs/13` 六条整体验收口径为准；证照或支付生产能力未就绪时，只能保持“仅展示”或沙箱模式。

## 15.7 并行关系与关键路径

- 产品/合规可在 A 阶段同步办理证照、支付商务、模型授权和第三方选型；这些工作不阻塞前端 Mock 原型。
- A1 完成后，A2 与 A3 可并行；A4 依赖审核、商品、账户和订单契约稳定；A5 可与 A4 后半段并行。
- B1 依赖 B0；B2 依赖账户、订单和金额模型；B3 依赖稳定的订单/争议事件；B4 依赖全部生产集成。
- 关键外部依赖：易宝进件与沙箱、newapi 版本/许可证、模型转售授权、KYC/电子签渠道、BSN-DDC 账号与备案口径。

## 15.8 交付与变更规则

- 每个阶段单独分支、分步提交，合并前必须通过对应验收门槛；
- 页面只通过 hooks/service 访问数据，不直接读取 Mock 或第三方 SDK；
- 金额用最小货币单位保存，状态转换在共享领域层集中校验；
- 第三方回调一律验签、幂等、可重放审计；
- 新需求先映射到现有 Must / Should / Could，不能绕过合规红线和阶段范围进入开发。
