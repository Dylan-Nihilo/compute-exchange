# 14 · 区块链信任基础设施（开发规格）

> 需求编号：`REQ-H-xxx`。对应汇报版第六章。
> 选链：BSN-DDC/文昌链（国家级开放联盟链），年成本 500–2,000 元。
> 接入方式：RESTful API，异步上链，不阻塞主业务流程。

---

## 14.1 定位与边界

区块链在平台中的角色：**不可篡改的存证层**——关键信用数据（机房履约、订单节点、违规记录）的 hash 存到链上，供第三方独立验证。

边界：
- ✅ 存证：hash + timestamp + 多方签名
- ✅ 验证：链上 hash 比对
- ❌ 不做：代币、DeFi、链上交易、智能合约自动化执行

---

## 14.2 数据写入规则（三条铁律）

### REQ-H-001 (P0) 单方不签名
交易双方主张的订单与交付事实不得由单一利益相关方人工写入，必须多方签名：
- 机房签名 = "我认可这条数据"
- 买家签名 = "我确认这条数据"
- 平台签名 = "我作为中立方见证了这条数据"

系统客观采集的 SLA、隔离风控系统的裁定和平台费率快照可由对应隔离系统密钥单签，但必须保留来源证据与审计链。

### REQ-H-002 (P0) 争议不上链
买卖双方对事实有分歧时，不存任何一方的说法。先走争议处理，只将**仲裁结论**上链。

### REQ-H-003 (P0) 违规必留痕
风控确认的恶意行为（虚假交付、刷单、挖矿），违规结论**强制上链且不可跳过**——运营无权阻止违规记录写入。

---

## 14.3 逐类数据的防恶意机制

| 数据 | 上链内容 | 签名方 | 防恶意写入机制 | 触发时机 |
|------|----------|--------|---------------|----------|
| **订单创建** (REQ-H-010) | hash(订单ID + 买家ID_hash + 机房ID_hash + 商品规格 + 时间戳) | 平台 + 买家 | 必须关联真实易宝支付流水；异常模式检测（同一买卖对高频小额订单） | 下单成功 |
| **交付确认** (REQ-H-011) | hash(订单hash + 交付凭证hash + 机房签名 + 买家确认或系统确认签名 + 平台见证签名) | 买家确认时三方签名；无争议超时时由系统代确认 | 48h 内买家确认则签收；未确认且未发起争议则系统自动确认；期限内发起争议则暂停自动确认与分账并进入工单 | 买家确认 / 无争议超时系统确认 |
| **SLA 履约率** (REQ-H-012) | hash(供给方ID + 周期 + SLA达成率 + 探测数据hash + 平台签名) | **平台单签**（基于系统自动采集） | SLA 不由机房自报——平台定期探测采集；每月更新前公示 3 天 | 每月更新 |
| **SLA 故障事件** (REQ-H-013) | hash(事件ID + 故障起止 + 探测证据hash + 处置结论 + 平台签名) | **平台单签** | 探测到不可用→自动生成事件；买家举报→人工核查后追加 | 故障关闭时 |
| **风控违规** (REQ-H-014) | hash(违规ID + 违规类型 + 处置结论 + 风控签名) | **风控系统自动签名**（人工不可跳过） | 运营无法跳过上链；申诉成功后替代结论上链但原记录 hash 不可逆 | 风控处置完成时 |
| **佣金规则快照** (REQ-H-015) | hash(平台费率配置 + 生效时间) | 平台签名 | 证明某时间段没偷偷改规则 | 费率变更时 |

## 14.4 签名实现

| 签名角色 | 密钥管理 | 触发方式 |
|----------|----------|----------|
| 买家签名 | 用户登录态 session 派生，或前端签名组件 | 用户主动操作（确认签收） |
| 机房签名 | 同上 | 机房操作（回填凭证） |
| 平台见证签名 | 服务端专用密钥，密钥管理服务存储，不入库 | 后端自动 |
| 系统/探测签名 | 同上 | 后端定时任务自动 |
| 风控系统签名 | 独立密钥，运营无访问权限 | 风控模块自动 |

签名算法：SM2（国密）或 ECDSA secp256k1（兼容 ETH 生态），以 BSN-DDC 最终支持为准。

## 14.5 系统架构

```
主业务（下单/交付/风控）
        │
        ▼ 事件发出（不等待）
┌───────────────────┐
│  存证消息队列      │  ← 异步解耦，链故障不丢消息
│  (Redis/RabbitMQ) │
└───────┬───────────┘
        ▼
┌───────────────────┐
│  存证服务 Worker   │  ← 计算 hash → 调 BSN API → 写回 TX ID
│  (独立进程)        │     失败重试 ×3 → 死信队列告警
└───────┬───────────┘
        ▼
┌───────────────────┐
│  BSN-DDC 文昌链    │  ← RESTful API
└───────────────────┘
```

### REQ-H-020 (P0) 异步上链
下单/交付/风控操作 → 同步返回成功（毫秒）→ 后台异步队列上链（几秒~十几秒）。链故障不影响主业务。

### REQ-H-021 (P0) 故障降级
BSN 不可用时：消息队列堆积 → 重试 3 次 → 失败进死信 → 告警通知运维 → 恢复后补推。

### REQ-H-022 (P0) TX ID 回写
上链成功后，TX ID 写回业务表（订单/供给方/风控记录），前端可直接查询。

## 14.6 验证接口

### REQ-H-030 (P0) 验证接口
```
GET /api/blockchain/verify?type=order&id=xxx
→ 返回：{ verified: true/false, txId: "0x...", chainTimestamp: "...", verifyUrl: "https://..." }
```

验证逻辑：
1. 从平台数据库取原始数据
2. 按同样规则计算 hash
3. 调 BSN 查询接口比对 hash
4. 匹配 = verified true
5. 返回 BSN 区块链浏览器链接供用户自行查验

### REQ-H-031 (P1) 前端验证入口
- 机房详情页：履约率旁 ⛓ 图标 → 点击弹出"链上可查"panel
- 订单详情页：存证时间线，每个关键节点可点击查验
- 机房工作台："信用档案已上链"提示

## 14.7 刷单与串通检测（补充风控）

### REQ-H-040 (P1) 刷单检测
- 同一买家-机房对短期内大量小额完美订单 → 标记
- 同一 IP/设备/支付账户关联多个买卖身份 → 标记
- 确认为刷单 → 违规标记强制上链 + 信用分降权 + 商品降权

### REQ-H-041 (P1) 运营抽检
随机抽样已完成订单，要求机房提供对应时段的真实算力使用日志。不能提供 → 存疑标记。

## 14.8 BSN-DDC 接入清单

| 步骤 | 内容 |
|------|------|
| 1 | BSN 开放平台注册 + 实名认证 |
| 2 | 申请 API Key + 密钥配置 |
| 3 | 开通存证合约（使用 BSN 官方标准存证合约模板） |
| 4 | 开发存证服务 Worker（调 REST API 存证/查询） |
| 5 | 联调：存证 → 返回 TX ID → 区块链浏览器可查 |
| 6 | 区块链信息服务备案（网信办，BSN 商务协助，非许可制） |

### 开发量估算
- 存证服务 Worker：1 人 × 2–3 天
- 业务事件对接：1 人 × 2–3 天
- 前端验证入口：1 人 × 1–2 天
- **合计约 1 周**

## 14.9 数据库表（新增）

```sql
-- 链上存证记录表
CREATE TABLE blockchain_attestations (
    id BIGINT PRIMARY KEY,
    target_type VARCHAR(32) NOT NULL,   -- order/delivery/sla/reputation/violation/commission_rule
    target_id VARCHAR(64) NOT NULL,      -- 业务表主键
    data_hash VARCHAR(128) NOT NULL,     -- 上链的 hash 值
    signers JSONB NOT NULL,              -- 签名方列表 [{role, signature, timestamp}]
    chain_tx_id VARCHAR(128),            -- BSN 返回的交易 ID
    chain_status VARCHAR(16) DEFAULT 'pending', -- pending/confirmed/failed
    created_at TIMESTAMP DEFAULT NOW(),
    confirmed_at TIMESTAMP
);
CREATE INDEX idx_attestation_target ON blockchain_attestations(target_type, target_id);
```
