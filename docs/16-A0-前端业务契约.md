## 16.1 已冻结决策

- v1 是响应式 Web，不开发 App 或小程序。
- 当前交付先完成高保真前端与 Mock 数据闭环；真实后端和第三方服务后接。
- 买家按订单直接支付，不提供现金余额或预充值。
- Token 额度是已购模型调用服务配额，不具有现金价值。
- 供给方手动开通算力并回填交付凭证，预留未来 API 接口。
- 买家确认交付或超时自动确认后触发延迟分账。
- 合规开关包含仅展示、限额真实交易、全量真实交易三态；限额模式允许支付，但受单笔和累计额度限制。
- 品牌未定前使用 semantic design tokens 驱动的中性主题。

## 16.3 核心状态机

### 资质

`draft → pending → reviewing → approved / rejected / changesRequested`；`rejected / changesRequested → pending` 可补件重提；`approved → frozen`；解冻后回到 `reviewing`。

### 算力商品

`draft → pending → onSale / rejected`；驳回后可重提；在售商品可进入 `soldOut`、`offline` 或 `disabled`。

### 算力订单

订单采用正交状态，避免退款、争议或冻结覆盖履约进度：

- 履约：`awaitingPayment → awaitingProvisioning → provisioning → awaitingAcceptance → active → completed`，各阶段可按规则取消；
- 支付：`pending → paid → refundPending → refunded`；退款失败可从 `refundPending` 回到 `paid`；
- 签收：`pending → buyerAccepted / timeoutAccepted`；发起争议时原子进入 `disputed`，买家胜诉后进入 `rejected`，供给方胜诉后进入 `buyerAccepted`；
- 风控：`clear ↔ frozen`；冻结不改写其他状态，但暂停履约推进、退款发起和分账；
- 争议：`none → open → resolvedForBuyer / resolvedForSupplier`。

单状态结构转换由 `canTransition` 校验；订单动作还必须通过 `canApplyOrderTransition` 聚合校验支付、交付凭证、签收、争议和风控条件。争议开启与裁决由 `applyOrderEvent` 原子更新签收、争议和履约状态，页面不得直接改写状态。

### 分账

`pending → delayed → processing → succeeded → reconciled`；失败后只允许重试进入 `processing`。支付后先创建 `delayed` 记录，只有买家确认交付或无争议超时、且风控状态为 `clear` 时才能进入 `processing`。

### 居间线索

`new → unassigned → following → quoted → won / closed`。

### 合规开关

`showcase ↔ limited ↔ open`，并允许从 `open` 紧急回退到 `showcase`。

## 16.4 权限原则

- 访客可浏览商品与提交融资留资，不能下单。
- 已认证 buyer 可下单；supplier、vendor、funder 自购时必须切换到已解锁的 buyer 身份，路由始终按当前身份判权。
- supplier 通过资质审核后才能发布算力商品。
- vendor 通过企业资质审核后才能发布设备或服务。
- operator 只能使用明确授予的审核、订单、财务、风控、内容或用户管理权限点。
- admin 独占 RBAC 配置和合规开关；具备 `manageUsers` 的 operator 可检索、冻结和解冻用户，不能修改权限模型。

完整矩阵保留在 `docs/02 §2.4`，运行时结果由 `accessFor` 提供。

## 16.6 测试边界

- `canTransition`：验证状态合法性，阻止跳过审核、履约或分账阶段。
- `canApplyOrderTransition`：验证支付、交付凭证、签收、争议、风控和分账的聚合守卫。
- `applyOrderEvent`：原子开启或裁决争议，防止产生不可达的中间订单快照。
- `accessFor`：验证角色对业务能力的允许、条件允许或拒绝。
- `routes`：验证路由唯一性和高权限页面隔离。
- Zod schema：拒绝非法订单快照、未备案自部署模型及缺少签名或链上凭证的存证记录。
- 后续页面测试只从用户可见动作和 service 接口观察结果，不测试组件内部实现。
