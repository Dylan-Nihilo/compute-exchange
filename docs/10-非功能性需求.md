# 10 · 非功能性需求（NFR）

> 需求编号：`REQ-N-xxx`。v1 Web 平台，移动端做响应式适配。

## 10.1 性能
- **REQ-N-001 (P0)** 门户/列表页首屏 ≤ 2.5s（正常网络），核心接口 P95 ≤ 500ms。
- **REQ-N-002 (P0)** 商品检索/比价支持分页与索引，万级商品流畅。
- **REQ-N-003 (P1)** 支付/分账等关键交易接口具备幂等与重试，异步回调最终一致。
- **REQ-N-004 (P1)** 余量库存高并发下单不超卖（锁/乐观并发控制）。

## 10.2 可用性与可靠性
- **REQ-N-010 (P0)** 核心交易链路可用性 ≥ 99.5%。
- **REQ-N-011 (P0)** 支付/分账/Token 充值具备失败重试、对账补偿、告警。
- **REQ-N-012 (P0)** 数据库定时自动备份（日备+周备，参考既有实践），可恢复演练。
- **REQ-N-013 (P1)** 关键依赖（易宝、newapi）故障降级与熔断、友好提示。

## 10.3 安全
- **REQ-N-020 (P0)** 满足**等保三级**技术要求（详见第11章控制项映射）：身份鉴别、访问控制、安全审计、入侵防范、数据完整性/保密性。
- **REQ-N-021 (P0)** 传输加密（全站 HTTPS/TLS）、敏感数据存储加密（身份证、银行卡、API Key、交付凭证）。
- **REQ-N-022 (P0)** 密钥管理：支付私钥、newapi 密钥走密钥管理服务，不入库不入前端不进日志。
- **REQ-N-023 (P0)** 防护：SQL 注入、XSS、CSRF、越权（IDOR）、文件上传校验、接口鉴权与限流。
- **REQ-N-024 (P0)** 登录/敏感接口按 IP/账号限流防撞库（参考既有 newapi 限流经验，反代须透传真实 IP）。
- **REQ-N-025 (P1)** 个人信息保护合规（《个人信息保护法》）：最小必要收集、授权、隐私政策、数据删除/导出。

## 10.4 可维护性 / 可扩展性
- **REQ-N-030 (P0)** 模块化设计（核心层 + 可扩展业务层），各板块可独立演进（呼应用户既往"模块化架构"偏好）。
- **REQ-N-031 (P1)** 配置化：费率、规则、模板、字典可后台配置，减少改代码。
- **REQ-N-032 (P1)** 多端预留：账户/权限体系为后续 App/小程序预留 API。
- **REQ-N-033 (P1)** i18n 结构预留（v1 仅中文）。

## 10.5 数据
- **REQ-N-040 (P0)** 数据分域：算力交易、Token、资金、风控日志分库/分表或逻辑隔离（Token 财务与算力财务必须分离，业务硬要求）。
- **REQ-N-041 (P0)** 数据留存：审计/交易/资金日志 ≥ 6 个月（满足网络安全法/监管）。
- **REQ-N-042 (P1)** 数据出境：v1 数据境内存储，不涉及出境（除非代销上游为境外，需评估，见第11/12章）。

## 10.6 兼容与可访问性
- **REQ-N-050 (P0)** 主流浏览器（Chrome/Edge/Safari/Firefox 近两年版本）兼容；响应式适配移动端。
- **REQ-N-051 (P1)** 基础可访问性（语义化、对比度、键盘可达）。

## 10.7 SEO / 运营
- **REQ-N-060 (P1)** 门户 SEO：SSR/预渲染、meta、sitemap、结构化数据（利于获客）。
- **REQ-N-061 (P1)** 埋点与转化分析（留资/下单漏斗）。

## 10.8 部署与运维
- **REQ-N-070 (P0)** 容器化部署（参考既有 docker compose 实践），独立 newapi 实例隔离部署。
- **REQ-N-071 (P1)** 灰度/合规开关由配置驱动（第8章 REQ-F-070）。
- **REQ-N-072 (P1)** 监控告警（服务健康、交易成功率、分账成功率、备案/资质到期）。
