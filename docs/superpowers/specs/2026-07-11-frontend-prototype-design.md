# 前端高保真原型 · 设计文档

> 日期：2026-07-11 ｜ 状态：前端实施决策已评审通过（Dylan）
> 范围：算力撮合交易平台 v1 的**前端设计阶段**——基于 HeroUI Pro 的高保真可交互原型 + Mock 数据层。后端、支付对接、区块链上链均不在本阶段范围。

## 1. 背景与目标

需求文档（docs/01-14）已评审，前端实施所需决策已冻结（见 §2）；Q-06、Q-15~19、Q-21 仍为商务或品牌挂起项，不阻塞 Mock 原型。当前阶段只做前端：产出一个完整可点击、数据"活"的原型，既能对外演示（客户/投资人/合规方），又作为后续正式开发的直接地基——接真 API 时只替换数据层实现，页面与组件不重写。

上线时间对齐合规时间线 2026-12（docs/11 §11.10 T3）。

## 3. 技术栈与总体架构

| 层 | 选型 |
|----|------|
| 框架 | Next.js 15（App Router）+ TypeScript（strict） |
| 样式 | Tailwind CSS v4 |
| 组件 | `@heroui/react` + `@heroui-pro/react`（复合组件模式，onPress） |
| 服务端数据 | TanStack Query + service 接口层 |
| 客户端状态 | Zustand（当前身份、UI 偏好、演示时钟） |
| 表单校验 | zod（schema 未来复用为 API 校验） |
| 测试 | Playwright 冒烟（4 条关键路径） |

**数据层约定**（原型即地基的关键）：

```
组件 → hooks（useOrders…）→ services 接口（OrderService…）→ mock 实现
                                                      ↘ 未来: http 实现
```

页面只碰 hooks，不感知数据来源。权限用 `usePermission` 钩子 + 路由守卫，按 docs/02 §2.4 权限矩阵实现，mock 阶段即生效。

## 4. 信息架构与路由

```
app/
├─ (portal)/            门户（批次4，SSR）：首页、compute/token/broker/leasing 板块页、留资、合规展示、存证验证
├─ market/              交易所（批次3）：检索比价、[productId] 详情+试算、checkout 模拟支付
├─ console/             工作台（批次2，统一外壳+身份切换器）
│   ├─ buyer/           待办、订单（凭证/签收/退款）、Token（额度/Key/用量）、账单发票
│   ├─ supplier/        资质、商品（时段/余量/调价）、接单履约（回填凭证）、结算中心
│   ├─ vendor/          设备发布、居间线索、佣金台账
│   └─ funder/          融资线索查看/认领
├─ admin/               运营后台（批次1）
│   ├─ 总览看板、review/（机房资质/商品/内容）、orders/、finance/（分账规则/对账/结算）
│   ├─ crm/、risk/（告警/冻结处置）、tokens/（模型/套餐）、cms/、users/（RBAC）
│   └─ settings/（合规开关）
└─ auth/                登录注册、KYC 向导（个人实名/企业认证）
```

## 5. 数据模型与 Mock 引擎

核心实体（`lib/types/`，即未来 API 契约草稿）：Account（多身份+各自认证状态）、SupplierQualification（资质台账+到期预警）、ComputeProduct（GPU/交付形态/计费/可售时段/余量/合规承诺）、Order（docs/02 §2.6 完整状态机+延迟分账）、SplitRecord（三方账目）、TokenPackage/TokenAccount、Lead（三类线索）、RiskAlert、Attestation（存证 mock txId）、ComplianceGate（三态全局生效）。

Mock 引擎：
- 种子数据：约 30 机房 / 80 商品 / 200 订单 / 50 线索，分布于各状态
- 动作驱动流转：admin 审核通过 → 供给方即可上架 → 买家可下单，跨端联动
- 可选时钟加速：演示模式下订单数分钟走完生命周期
- localStorage 持久化，一键重置；支持注入延迟/失败以验收三态

## 6. 视觉方向

- 基调：B2B 交易基础设施——专业、可信、数据密度高（Stripe/Linear 式克制）；门户批次 4 单独定调
- 主题：HeroUI Pro semantic design tokens；品牌未定前使用中性主题，按 background / surface / text / border / action / status 定义颜色角色，业务组件不写死品牌色，明暗双主题
- 字体：思源黑体/系统栈（可商用）；金额/余量 tabular-nums
- 组件模式：四套状态徽章色板（订单/审核/分账/风控语义统一）、分账三方拆解可视化、资质有效期进度条、⛓ 存证徽章
- 实现前置：每批次开工先过 `heroui-pro-design-taste` 与 `heroui-react-pro` skill

## 8. 错误处理与测试

- 每个列表/详情页强制三态：loading 骨架屏 / empty 含引导 / error 可重试
- 长表单 zod 校验（KYC、上架、留资）
- Playwright 冒烟 4 条路径：①全闭环（审核→上架→下单→签收→分账展示）②身份切换权限 ③合规开关三态影响 ④Token 购买→Key→账单
- TS strict、ESLint、路由级代码分割

## 9. 明确不做（本阶段）

真实后端/数据库、易宝对接（模拟支付页代替）、newapi 对接（mock Token 数据）、BSN 上链（mock txId）、电子签/实名/短信等第三方、单元测试全覆盖、多语言。
